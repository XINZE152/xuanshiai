# 项目代码修改规则

> 本文件是 Codex 和 Claude Code 共用的项目规则正文。项目负责人可以直接在下方补充、修改或删除规则。
这是为您后端项目定制的 Codex & Claude Code 编程规则集，涵盖了接口管理、安全校验、敏感信息处理和对话总结等关键环节。

---
# 后端项目 Codex & Claude Code 协作编程规则 v1.1


## 一、通用原则

1. **安全第一**：任何功能实现必须优先考虑安全性，禁止为了快速实现功能而省略校验、过滤或加密步骤。

2. **文档即代码**：接口文档与代码具有同等重要性，任何接口变更必须同步更新文档。

3. **可追溯性**：每次对话结束时的总结必须清晰、结构化，便于团队审查和前端对接。

4. **最小权限原则**：敏感信息（密码、密钥、Token）不得硬编码，必须通过环境变量或配置中心注入。


## 二、接口管理规则（前端可见）

> **目的**：确保前端团队能实时、准确地获取接口定义，减少联调障碍。

### 2.1 新增接口

**必须做**：
1. 在项目 `docs/api/` 目录下创建或更新对应的接口文档（推荐 OpenAPI/Swagger 格式）。
2. 文档必须包含：**URL**、**Method**、**Headers**、**Request Body/Params 示例**、**Response 示例**、**错误码说明**。
3. 在代码中添加对应的路由和 Controller 层，并确保入参校验逻辑完整。
4. 在本次对话的【改动总结】中，以 **"新增接口"** 为标题，列出接口全称和主要用途。

**禁止做**：
- 不写文档直接让前端"看代码"。
- 接口参数定义模糊（如 `data` 字段不说明内部结构）。

### 2.2 修改接口（包括入参、出参、路径、方法）

**必须做**：
1. 同步更新接口文档，并高亮标记 **变更内容**（如 `// changed` 或使用 OpenAPI 的 `deprecated` + 新版本）。
2. 若为破坏性变更（Breaking Change），必须在文档中注明 **迁移指南** 或提供过渡方案。
3. 在本次对话的【改动总结】中，以 **"修改接口"** 为标题，说明变更点及影响范围。

**禁止做**：
- 只改代码不改文档。
- 删除或重命名旧字段而不提供兼容处理或通知前端。

### 2.3 删除接口

**必须做**：
1. 先确认该接口无任何调用方（可通过日志或调用链分析）。
2. 在文档中标记为 `deprecated` 至少一个版本周期后再移除。
3. 删除代码路由和相关 Handler。
4. 在本次对话的【改动总结】中，以 **"删除接口"** 为标题，注明删除原因和替代方案（如有）。

**禁止做**：
- 直接删除代码而不通知团队。
- 删除后不更新文档导致前端引用失效。


## 三、安全与校验规则

> **核心**：接口不能只实现业务功能，必须内置安全防御。

### 3.1 输入校验（防注入、防篡改）

**所有用户输入**（包括 Path Params、Query Params、Body、Headers）必须进行：
- **类型校验**（如 `int`、`string`、`email`）。
- **长度/范围校验**（防止溢出或 DoS）。
- **格式校验**（如 UUID、日期格式、枚举值）。
- **业务校验**（如订单金额不能为负、状态流转必须合法）。

**实现方式**：使用 Validation 框架（如 `@Valid`、`Joi`、`pydantic`）并定义清晰错误信息。

### 3.2 敏感信息处理

**绝对禁止**：在代码、日志、注释、返回值中泄露真实密码、密钥、Token。

**必须**：
- 所有密钥/密码通过环境变量 `process.env` 或配置中心读取。
- 日志中对于密码、身份证号等字段进行脱敏（如 `******`）。
- 返回给前端的响应中，移除密码哈希、密钥等字段（可在全局序列化层过滤）。

### 3.3 接口权限与校验

**每个接口必须明确权限要求**（如 `@Permission("user:read")`）。

**关键操作**（删除、修改、资金操作）必须进行：
- **身份校验**（Token/Session 有效性）。
- **操作权限校验**（当前用户是否拥有该资源操作权限）。
- **幂等性控制**（如使用 Idempotent-Key 防止重复提交）。


## 四、敏感信息填写指引

> 当新功能或修改涉及以下信息时，**必须** 告知用户自行填写，绝不可擅自填充示例值或默认生产值。

**涉及内容**：
- 数据库连接串（`DB_URL`）
- 第三方 API Key / Secret（如支付、短信、OSS）
- JWT Secret / 加密 Salt
- 管理员初始密码/邮箱
- Redis / RabbitMQ 连接密码

**告知方式**：在对话总结中明确写出：
> **【敏感信息填写位置】**：
> 请在 `./config/.env.example` 中找到 `YOUR_XXX_KEY` 变量，将其替换为您实际的值。该文件不会提交至 Git，请妥善保管。


## 五、每次对话结束的【改动总结】模板

> **必须** 在每次对话最后，按以下格式输出总结，方便前端和其他成员查阅。

````markdown
### 📋 本次对话改动总结（YYYY-MM-DD）

#### 1. 新增接口
- `POST /api/v1/orders` - 创建订单
  - 用途：用户下单
  - 文档更新：`docs/api/orders.yaml#L12-L45`

#### 2. 修改接口
- `GET /api/v1/users/{id}` 
  - 变更：响应中新增 `userLevel` 字段，移除 `score` 字段（已废弃）
  - 文档标记：`deprecated` 并指向新版本 `v2`

#### 3. 删除接口
- `DELETE /api/v1/products/{id}/force` （硬删除）
  - 原因：改用软删除 `PUT /api/v1/products/{id}/status`
  - 替代方案：请使用软删除接口

#### 4. 安全与校验增强
- `POST /api/v1/payments` 增加幂等性校验（`Idempotent-Key` Header）
- 所有 `String` 类型入参增加 XSS 过滤（使用 `xss-filter` 中间件）

#### 5. 敏感信息填写位置
- 支付宝支付回调密钥：请在 `config/production.env` 中配置 `ALIPAY_APP_SECRET`
- 数据库读写分离密码：请在 `config/.env` 中配置 `DB_REPLICA_PASSWORD`

#### 6. 其他变动
- 升级 `express` 从 `4.18` 至 `4.19`（安全补丁）
- 统一错误码 `E10001` 改为 `E_INVALID_PARAM`
````


## 六、技术方案可行性评判规则（前置必做）

> **背景**：用户可能对技术栈、架构或实现细节了解不深入，提出的技术方案可能存在隐性风险（如性能瓶颈、安全漏洞、维护困难、与现有系统不兼容等）。**必须**先评判、再执行，避免"南辕北辙"。

### 6.1 评判范围

当用户提出以下内容时，**必须先进行可行性评判**：

- 整体技术选型（如"用 MongoDB 替换 MySQL"、"改用微服务"）
- 架构设计方案（如"用消息队列解耦"、"引入缓存层"）
- 第三方服务/中间件引入（如"接入某某支付"、"用 Redis 做分布式锁"）
- 数据库设计变更（如"改表结构"、"分库分表"）
- 性能优化策略（如"用多线程并发处理"）
- 安全方案（如"用 JWT 替代 Session"）
- 业务逻辑实现路径（如"用定时任务批量处理订单"）

### 6.2 评判标准（五维度）

对每个技术方案，从以下五个维度逐一评估：

| 维度 | 评估要点 |
|------|----------|
| **技术可行性** | 当前技术栈是否支持？是否有现成库/框架？是否需要大量造轮子？ |
| **安全性** | 是否引入新的攻击面（如注入、越权、数据泄露）？是否涉及敏感信息处理？ |
| **性能影响** | 是否会导致响应延迟增加？是否会造成资源瓶颈（CPU/内存/IO）？是否符合 SLA？ |
| **维护成本** | 是否增加系统复杂度？团队成员是否熟悉？未来扩展是否困难？ |
| **兼容性** | 是否影响现有功能？是否需要数据迁移？前端/下游是否需要同步改动？ |

### 6.3 评判结果与处理方式

评判后必须给出明确的结论和后续行动：

#### ✅ 可行（可直接执行）
- 给出明确的"可行"结论，并简述依据（如"该方案在当前技术栈下已有成熟实现，无安全风险"）。
- 然后按原规则执行编码。

#### ⚠️ 有条件可行（需调整后执行）
- 明确指出**需要修改的部分**，并给出具体建议。
- 示例："该方案基本可行，但建议将 `Redis` 作为缓存而非消息队列使用，因为您的场景是读多写少。调整后即可执行。"
- 等待用户确认调整方案后再执行。

#### ❌ 不可行（建议否决）
- 明确说明不可行的原因（引用具体技术依据）。
- 提供 **替代方案**（至少一个），并解释为什么替代方案更优。
- 示例："使用 `悲观锁` 处理并发订单不推荐，因为在高并发下会导致大量连接阻塞。建议改用 `乐观锁` 或 `分布式锁`。如果您同意，我将按新方案实现。"

### 6.4 评判输出格式（必须）

在每次技术方案评判后，必须按以下格式输出：

````markdown
### 🔍 技术方案可行性评判

**用户提出的方案**：[简述，如"用 RabbitMQ 处理订单超时关闭"]

**评判结果**：⚠️ 有条件可行 / ✅ 可行 / ❌ 不可行

**详细分析**：
- **技术可行性**：[评估结果]
- **安全性**：[评估结果]
- **性能影响**：[评估结果]
- **维护成本**：[评估结果]
- **兼容性**：[评估结果]

**结论与建议**：
[具体说明建议，如需调整则写明调整步骤，如需否决则写明替代方案]

**下一步操作**：
- 若可行 → 进入编码阶段
- 若需调整 → 等待用户确认后再执行
- 若否决 → 提供替代方案并等待用户决策
````

### 6.5 特殊场景处理

| 场景 | 处理方式 |
|------|----------|
| 用户方案明显存在安全漏洞 | 直接给出 **❌ 不可行** 结论，并强调安全风险，提供安全替代方案 |
| 用户方案成本过高（如引入全新中间件） | 给出 **⚠️ 有条件可行**，建议先用现有工具实现，并说明引入新中间件的代价 |
| 用户方案与现有架构严重冲突 | 给出 **❌ 不可行**，详细解释冲突点，并推荐兼容性更好的方案 |
| 用户坚持采用不推荐的方案 | 记录风险提示，明确告知后果（如性能下降、安全风险），若用户仍坚持，则在代码中添加 `// TODO: 需重构` 注释并继续实现，但必须在总结中高亮风险 |


## 七、功能新增/修改完整工作流

> 每次实现新功能或修改现有逻辑时，请按此 checklist 执行：

0. **🔍 评判用户技术方案**（按 6.3/6.4 格式输出评判结果，获取确认后进入下一步）

1. **分析需求**，明确输入、输出、异常场景。

2. **设计接口**（路径、方法、参数、响应）。

3. **编写接口文档**（先文档后代码，或同步更新）。

4. **实现业务逻辑**。

5. **加入校验**（输入、权限、幂等、敏感数据脱敏）。

6. **编写/更新单元测试**（覆盖正常、边界、异常情况）。

7. **检查敏感信息**：若有新增外部依赖，指引用户配置。

8. **代码审查自检**：
   - [ ] 是否有 SQL 注入风险？（使用 ORM/参数化查询）
   - [ ] 是否有敏感信息硬编码？
   - [ ] 日志是否安全？
   - [ ] 错误信息是否会暴露系统内部细节？

9. **生成最终改动总结**（按第五部分模板）。


## 八、特别约束（Codex / Claude Code 执行层）

- **禁止跳过安全检查**：如果用户要求关闭校验或"快速实现"，必须回复："出于安全考虑，我无法省略输入校验/敏感信息处理，请理解。"

- **环境隔离**：在操作数据库或缓存时，自动识别当前环境（dev/test/prod），并在非生产环境使用模拟数据或发出警告。

- **变更影响分析**：每次改动后，自动提示可能影响的现有接口或模块（如"此修改会影响 `/api/v1/users` 的排序逻辑"）。

- **技术方案评判优先级**：技术方案评判（第六章）优先级高于所有编码工作，未完成评判并得到用户确认前，不得开始任何代码编写。
